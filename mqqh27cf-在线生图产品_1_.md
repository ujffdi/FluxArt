# 在线生图产品 V1.0 混合规格文档

## 1. 文档定位与使用方式

### 1.1 文档定位
本文档不是纯 PRD，也不是纯技术设计， **PRD + 实施规格混合版**。它同时服务三类读者：

- 产品与设计：理解 V1 要做什么、为什么做、做到什么程度算完成
- 研发团队：明确页面、流程、接口、数据模型、状态与边界
- AI 开发代理：可以直接据此拆任务、补接口、落页面、编写验收用例

### 1.2 适用范围
本文档只约束 **在线生图产品 V1.0**，并且默认基于当前仓库技术栈落地：

- Web 应用：Next.js + React + TypeScript
- 状态管理：Zustand
- 服务端 / API 层：Next.js Route Handlers / Server Actions，运行在 Node.js Runtime
- AI 集成：Next.js 服务端图像模型接入层，默认 OpenAI `gpt-image-2`，支持配置自定义图像模型
- 数据层：Repository Adapter 边界；V1 本地默认 mock/in-memory，可选 SQLite 文件库，生产可替换为 MySQL/Postgres + Redis
- 对象存储：现有对象存储能力或兼容 OSS/MinIO 的实现

### 1.3 目标输出
V1 文档需要支持以下结果：

- 能指导前后端协作开发
- 能让 AI 编码代理按模块实施
- 能约束范围，避免把远期能力误做进 V1
- 能作为后续接口联调与验收依据

### 1.4 非目标
以下内容不属于本次 V1 必做范围：

- 社区、作品公开广场、点赞评论
- LoRA 训练、自定义模型训练
- ControlNet 细粒度控制
- 批量工作流编排
- 多供应商智能调度策略平台化
- 企业级 API 开放平台

## 2. 产品概述与 V1 范围

### 2.1 产品背景
AI 图像生成需求已从“能否生成图片”转向“能否稳定地产出可用图片并支持继续修改”。单次生成型产品存在三个核心问题：

- 结果不稳定，用户需要反复抽卡
- 生成后无法继续精修，只能推倒重来
- 权益与下载规则不清，难以形成商业闭环

### 2.2 产品定位
一句话定位：**一款面向个人创作者与中小企业的在线 AI 生图工具，支持从生成到继续编辑的完整工作流。**

核心价值：

- 让用户不仅能生图，还能继续改图
- 把结果图变成后续编辑入口，而不是终点
- 用积分与会员体系承接下载、高清、无水印等权益

### 2.3 V1 产品目标
V1 目标分为四类：

#### 业务目标
- 建立“生成 -> 编辑 -> 下载/付费”的基础闭环
- 验证用户是否愿意为无水印、高清、进阶能力付费

#### 用户目标
- 首次进入后可快速理解产品能力并完成第一次生成
- 生成后的结果图可以继续用于图生图、局部重绘、扩图

#### 技术目标
- 跑通第三方图像生成服务接入
- 建立任务记录、资产管理、积分扣减、会员鉴权基础链路
- 让前端可以统一查询任务状态与图片资产结果

#### 体验目标
- 标准单图生成目标耗时 `< 15s`
- 任务成功率目标 `> 95%`
- 用户可以理解失败原因，而不是只看到泛化错误

### 2.4 V1 范围
V1 聚焦以下四类能力：

#### A. 生图与编辑核心
- 文生图（T2I）
- 图生图（I2I）
- 局部重绘（Inpainting）
- 扩图（Outpainting）

#### B. 结果资产管理
- 任务记录
- 图片历史记录
- 单图详情
- 源图与衍生图版本链

#### C. 账户与权益闭环
- 登录鉴权
- 积分余额
- 会员状态
- 下载权限控制
- 无水印 / 高清权限判断

#### D. 商业基础能力
- 积分包下单接口占位
- 会员订阅下单接口占位
- 订单记录与权益生效状态

### 2.5 V1 不做内容
以下内容可保留为后续版本方向，但不纳入 V1 开发承诺：

- 完整商城化会员中心
- 复杂促销体系
- 多模型自由切换配置台
- 可视化工作流编排
- 公开作品分享社区

## 3. 用户角色与关键场景

### 3.1 目标用户

| 用户角色 | 典型特征 | 核心诉求 |
|---|---|---|
| 个人创作者 | 自媒体、插画爱好者、内容编辑 | 出图快、质量稳定、能继续微调 |
| 中小企业用户 | 电商运营、市场营销、品牌小团队 | 商品图/海报批量产出、无水印下载、可商用 |
| 试用用户 | 对 AI 生图感兴趣但未付费 | 快速体验、操作简单、知道付费价值 |

### 3.2 关键用户故事
- 作为内容创作者，我希望输入提示词即可生成配图，并能对不满意区域继续修改，以减少返工。
- 作为电商卖家，我希望上传商品图后生成不同风格版本，并保留原图与新图关系，方便筛选。
- 作为试用用户，我希望先免费体验核心能力，如果结果足够好，再付费解锁无水印或更多额度。

### 3.3 关键业务流程

#### 流程 1：游客试用到首次生成
1. 用户进入 AI 生图工作台
2. 浏览能力说明、示例图、权益差异
3. 输入提示词并尝试生成
4. 若策略允许试用则直接创建试用任务；否则引导登录
5. 登录后进入正式额度与权益体系

#### 流程 2：生成到结果确认
1. 用户提交参数
2. 系统校验登录状态、积分余额、会员权限、输入合规
3. 创建任务并进入异步处理
4. 任务完成后展示结果图列表
5. 用户可执行下载、再次生成、作为参考图继续生成、进入重绘、进入扩图

#### 流程 3：结果到继续编辑
1. 用户在结果图或历史资产中选择一张图片
2. 进入图生图 / 局部重绘 / 扩图模式
3. 发起新任务
4. 系统产出新版本图片，不覆盖原图
5. 新图与源图通过 `sourceAssetId` 建立关系，形成版本链

## 4. 页面信息架构与页面规格

### 4.1 页面清单

| 页面 | 路由建议 | 登录要求 | 页面目标 |
|---|---|---|---|
| AI 生图工作台 | `/workspace/image` | 建议登录，支持部分试用策略 | 承载文生图 / 图生图入口与结果浏览 |
| 图片编辑页 | `/workspace/image/edit/:assetId` | 必须登录 | 承载局部重绘与扩图 |
| 图片资产中心 | `/workspace/image/assets` | 必须登录 | 查看历史任务、图片资产、版本链 |
| 权益/购买弹层或页面 | 复用弹层，必要时独立 `/workspace/billing` | 必须登录 | 展示积分、会员、购买入口、权限说明 |

### 4.2 AI 生图工作台

#### 页面目标
统一承载四类模式中的前两类入口：

- 文生图
- 图生图

局部重绘和扩图可从此页进入说明态，但实际编辑应跳转至图片编辑页。

#### 页面布局建议
- 左侧：参数面板
- 中间：结果区 / 上传预览区 / 空态示例区
- 右侧：任务状态、最近生成记录、继续编辑入口

#### 核心组件
- 模式切换器
- Prompt 输入区
- Negative Prompt 输入区
- 风格预设选择器
- 尺寸选择器
- 数量选择器
- 参考图上传组件（图生图模式）
- 权益提示条（积分余额 / Pro 限制）
- 结果图网格
- 任务状态提示

#### 页面状态
- 初始空态
- 参数编辑态
- 提交中
- 任务处理中
- 结果展示态
- 权益不足态
- 合规拦截态
- 失败重试态

### 4.3 图片编辑页

#### 页面目标
承接基于已有图片继续加工的两类能力：

- 局部重绘
- 扩图

#### 核心组件
- 源图画布
- 画笔与蒙版工具（局部重绘）
- 扩展方向与比例设置（扩图）
- 编辑说明输入区
- 编辑结果预览
- 原图 / 新图切换与对比
- 版本链入口

#### 页面约束
- 编辑行为只针对已有资产发起，不支持孤立上传后直接进入编辑
- 编辑结果必须生成新的资产记录，不覆盖原资产

### 4.4 图片资产中心

#### 页面目标
统一查看：
- 任务列表
- 图片画廊
- 单图详情
- 下载记录
- 版本链

#### 页面能力
- 按任务类型筛选
- 按时间排序
- 查看任务状态与失败原因
- 进入单图详情
- 从任意结果图继续图生图 / 重绘 / 扩图

### 4.5 权益相关页面/弹层
V1 不要求搭建完整商城式页面，但必须具备以下能力：

- 显示积分余额
- 显示会员状态
- 展示当前功能是否受限
- 支持跳转到积分包/会员购买入口
- 在下载时说明水印、高清、商用权限差异

## 5. 功能模块实施规格

### 5.1 模块一：文生图（T2I）

#### 功能说明
用户输入提示词，系统生成一张或多张图片结果。

#### 输入参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `prompt` | string | 是 | 正向提示词，建议 10-300 字 |
| `negativePrompt` | string | 否 | 反向提示词 |
| `size` | enum | 否 | 默认 `1:1` |
| `count` | integer | 否 | 默认 `1`，V1 范围 `1-4` |
| `stylePreset` | enum | 否 | 默认 `general` |

#### 前端行为
- 未登录时依据试用策略提示试用或登录
- 生成前展示预计消耗积分
- 提交后进入任务处理中态
- 成功后展示结果图网格，并提供继续编辑入口

#### 后端职责
- 校验登录、试用资格、积分/会员权限
- 校验提示词合法性
- 创建任务记录并提交 AI 服务执行
- 任务完成后写入资产记录与积分流水

#### 规则约束
- 免费用户能力受每日次数或积分限制，具体策略由配置决定
- 生成结果需经过输出审核，审核通过后才进入可见资产
- 大尺寸或多图生成可增加积分消耗

#### 验收标准
- 用户可成功提交文生图任务并看到任务状态变化
- 成功结果进入历史资产
- 扣费与余额展示一致
- 失败时能看到明确失败原因

### 5.2 模块二：图生图（I2I）

#### 功能说明
用户上传参考图，并结合提示词生成风格变体或修改后的新图。

#### 输入参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `sourceImage` / `sourceAssetId` | file/string | 是 | 参考图文件或已有资产 ID |
| `prompt` | string | 是 | 修改方向说明 |
| `strength` | integer | 否 | 默认 `50`，范围 `0-100` |
| `structureMode` | enum | 否 | `balanced` / `outline` / `pose` / `color` |
| `size` | enum | 否 | 默认跟随源图或预设尺寸 |
| `stylePreset` | enum | 否 | 风格预设 |

#### 前端行为
- 上传前校验格式和大小
- 上传后展示预览图与参数区
- 权益不足时明确提示 `Pro` 或积分限制

#### 后端职责
- 审核上传图片
- 记录原始图片来源
- 调用 AI 服务发起图生图任务
- 生成结果入库为新资产

#### 规则约束
- 免费版上传与尺寸能力可受限
- 图生图积分消耗高于文生图
- 上传违规图片需阻断并记录

#### 验收标准
- 用户可基于上传图成功生成新图
- 上传违规图会被拦截
- 生成结果与源图建立可追踪关系

### 5.3 模块三：局部重绘（Inpainting）

#### 功能说明
用户在已有图片上圈定局部区域，并通过文字指令重新生成该区域内容。

#### 输入参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `sourceAssetId` | string | 是 | 源图片资产 ID |
| `maskImageUrl` / `maskData` | string | 是 | 蒙版数据 |
| `instruction` | string | 是 | 局部重绘指令 |
| `brushSize` | integer | 否 | 画笔大小 |
| `strength` | integer | 否 | 默认 `70` |

#### 前端行为
- 从已有资产进入编辑页
- 支持画笔涂抹与蒙版预览
- 支持原图 / 新图对比

#### 后端职责
- 校验源图资产归属
- 保存蒙版与编辑参数
- 调用 AI 服务执行局部重绘
- 生成新资产并建立版本关系

#### 规则约束
- 局部重绘结果不覆盖原图
- 编辑尺寸可压缩到系统允许范围，避免超出模型限制
- 违规编辑指令同样走输入审核

#### 验收标准
- 用户可以对历史图片完成局部重绘
- 新图为独立资产，原图仍可访问
- 版本链能看到来源关系

### 5.4 模块四：扩图（Outpainting）

#### 功能说明
用户基于已有图片扩展画布，系统补全新增区域。

#### 输入参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `sourceAssetId` | string | 是 | 源图资产 ID |
| `expandDirections` | string[] | 是 | `top` / `bottom` / `left` / `right` |
| `expandRatio` | enum | 否 | V1 固定 `1.5x` / `2x` |
| `instruction` | string | 否 | 可选补充说明 |

#### 前端行为
- 编辑页展示扩展预览区域
- 明确提示扩图为高消耗能力
- 非 Pro 时直接提示受限

#### 后端职责
- 校验扩图权限
- 根据方向与比例构建扩图请求
- 生成新资产并回写版本关系

#### 规则约束
- V1 仅支持固定比例，不支持任意自由拖拽扩展
- 默认仅面向 Pro 或高阶权益开放

#### 验收标准
- 用户可基于已有图完成扩图
- 扩图结果为独立新资产
- 受限用户会收到明确权益提示

### 5.5 模块五：图片资产中心

#### 功能说明
统一管理任务结果、图片历史、版本关系与下载行为。

#### 核心能力
- 任务列表
- 图片列表
- 单图详情
- 源图 / 衍生图关系展示
- 下载记录追踪

#### 验收标准
- 用户可以找到历史图片
- 用户可以查看某张图由哪张图衍生而来
- 用户可以从历史图继续发起编辑任务

### 5.6 模块六：账户与权益

#### 功能说明
V1 只保留闭环所需能力，不展开复杂运营系统。

#### 范围包含
- 登录状态识别
- 积分余额查询
- 会员状态查询
- 不同功能的权限判断
- 订单记录与权益生效状态

#### 权益规则示例
- 免费用户：基础生成、次数/积分受限、默认带水印
- Pro 用户：更高额度、无水印下载、扩图与高清能力开放
- 点数包用户：按量扣费，但是否享有无水印/商用能力由业务策略决定

#### 验收标准
- 前后端对权益判断一致
- 不能出现前端显示可用但后端拒绝且无解释的情况

## 6. 基于当前仓库的 Next.js 技术落地映射

### 6.1 前端落点
建议按当前项目结构扩展：

- `src/app/workspace/image/page.tsx`：新增 AI 生图工作台页面
- `src/app/workspace/image/edit/[assetId]/page.tsx`：新增图片编辑页
- `src/app/workspace/image/assets/page.tsx`：新增图片资产中心页
- `src/app/workspace/billing/page.tsx` 或弹层组件：承接权益 / 购买场景
- `src/features/image-generation/`：拆出图片参数面板、上传区、画布编辑区、结果网格、权益弹层
- `src/stores/`：新增或扩展图片生成相关 Zustand store，统一管理模式、当前任务、当前资产、权限提示

### 6.2 API 层职责
Next.js API 层通过 `src/app/api/**/route.ts` 或必要的 Server Actions 负责：

- 鉴权
- API 路由
- 限流
- 解析与校验请求参数
- 统一响应结构
- 将用户身份传递给服务端业务模块

不新增业务逻辑。

### 6.3 业务服务职责
`src/server/image/business/` 负责：

- 图片任务主记录
- 图片资产记录
- 积分扣减与流水
- 会员权益与订单记录
- 下载权限判断
- 历史列表与详情查询

业务服务不得直接依赖 mock 数据或具体数据库客户端；数据访问统一经由 `src/server/data/repositories.ts` 暴露的 repository 接口。当前 V1 本地实现使用 mock/in-memory adapter，后续接入 SQLite、MySQL、Postgres 或云数据库时，只替换 adapter，不改页面与 API 契约。

### 6.4 AI 服务职责
`src/server/image/ai/` 负责：

- 接第三方图像生成模型
- 默认使用 OpenAI `gpt-image-2` 模型
- 通过配置支持切换为其他自定义模型或自定义供应商
- 执行文生图 / 图生图 / 重绘 / 扩图
- 管理生成过程的状态更新
- 协同输入审核 / 输出审核流程

#### 模型接入策略
V1 不做复杂的多供应商智能调度平台，但需要预留清晰的模型适配边界：

- 默认供应商：`openai`
- 默认模型：`gpt-image-2`
- 默认配置项：`IMAGE_MODEL_PROVIDER=openai`、`OPENAI_IMAGE_MODEL=gpt-image-2`
- 自定义模型配置：允许通过环境变量或后台配置指定 `provider`、`model`、`baseUrl`、`apiKeySecretRef`
- 前端不直接感知模型供应商，只提交统一的生成 / 编辑参数
- 服务端统一将业务参数转换为具体供应商请求，并把模型调用结果转换为统一任务状态与资产记录
- 模型 Key、Base URL、第三方凭证只允许保存在服务端环境变量或密钥管理系统中

### 6.5 公共模块职责
`src/server/shared/` 与 `src/types/` 负责沉淀：

- DTO / VO / 枚举
- 任务状态枚举
- 资产状态枚举
- 权益判断响应结构
- 通用响应包装与错误码

### 6.6 服务边界原则
- 权限、积分、订单相关判断统一放 `src/server/image/business/`
- 模型调用、图像处理参数、第三方交互统一放 `src/server/image/ai/`
- 前端通过 Next.js API 路由访问统一 API，不感知服务端内部模块拆分

## 8. API 契约草案

所有对前端开放的接口统一通过 Next.js API 路由暴露，并沿用统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

### 8.1 任务接口

#### `POST /api/image/tasks`
创建图片任务。

请求体：

```json
{
  "taskType": "t2i",
  "prompt": "一只戴耳机的橘猫坐在电脑前",
  "negativePrompt": "模糊, 低质量",
  "size": "1024x1024",
  "count": 1,
  "stylePreset": "general",
  "sourceAssetId": null,
  "strength": null,
  "structureMode": null,
  "maskImageUrl": null,
  "instruction": null,
  "expandDirections": null,
  "expandRatio": null
}
```

返回体：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "taskId": 1001,
    "status": "queued",
    "chargedCredits": 1,
    "remainingCredits": 99
  }
}
```

#### `GET /api/image/tasks/{taskId}`
查询任务详情与状态。

#### `GET /api/image/tasks`
分页查询任务列表，支持按 `taskType`、`status`、时间范围筛选。

### 8.2 资产接口

#### `GET /api/image/assets`
分页查询图片资产列表。

#### `GET /api/image/assets/{assetId}`
查询单图详情，返回：
- 基本信息
- 任务摘要
- 可执行动作
- 版本链摘要
- 下载权限

#### `POST /api/image/assets/{assetId}/download`
申请下载。

返回：
- 下载地址或临时票据
- 是否带水印
- 是否高清

### 8.3 账户权益接口

#### `GET /api/account/credits`
获取积分余额与最近变化摘要。

#### `GET /api/account/membership`
获取会员状态、到期时间、可用权益。

### 8.4 订单接口

#### `POST /api/orders/credits`
创建积分包订单。

#### `POST /api/orders/membership`
创建会员订阅订单。

V1 允许先完成接口占位与状态记录，不强制在本阶段完成完整支付集成。

### 8.5 错误码语义建议
至少覆盖以下业务语义：

- `UNAUTHORIZED`
- `TRIAL_LIMIT_REACHED`
- `INSUFFICIENT_CREDITS`
- `PRO_REQUIRED`
- `PROMPT_BLOCKED`
- `IMAGE_UPLOAD_BLOCKED`
- `TASK_TIMEOUT`
- `GENERATION_FAILED`
- `DOWNLOAD_FORBIDDEN`

## 9. 非功能需求与验收标准

### 9.1 性能要求

| 指标 | 目标 |
|---|---|
| 标准单图生成耗时 | P90 `< 15s` |
| 页面首屏加载 | `< 2s` |
| 生成成功率 | `> 95%` |
| 并发能力 | 支持 50+ 用户并发提交任务 |

### 9.2 安全与合规要求
- 输入提示词必须经过敏感词过滤
- 上传图片必须经过基础违规检测
- 输出图片必须支持审核流程
- 模型 Key 与第三方凭证仅允许后端持有
- 用户上传图片不得用于模型训练
- 需要有清晰的隐私与资源保留说明

### 9.3 可用性要求
- 任务超过设定时长必须进入超时失败态
- 失败任务必须带失败原因
- 任务状态变化需要可追踪
- 资产、任务、积分流水、下载行为要能关联审计

### 9.4 功能验收清单

#### 文生图
- 输入提示词后可创建任务并成功返回图片
- 多尺寸与多数量策略按配置生效
- 扣费正确

#### 图生图
- 可上传参考图并发起任务
- 免费与 Pro 限制生效
- 上传违规图被阻断

#### 局部重绘
- 可对已有资产进行涂抹编辑
- 生成后形成新资产，不覆盖原图
- 能看到原图与新图关系

#### 扩图
- 可对已有资产进行扩图
- 扩图能力权限正确生效
- 输出为独立新资产

#### 资产中心
- 可查看历史任务、图片、版本链
- 可从历史图继续进入编辑

#### 权益体系
- 积分不足会提示
- Pro 限制会提示
- 下载权限与实际结果一致

## 10. 开发边界与未决问题

### 10.1 开发边界
本次 V1 重构默认要求：

- 基于当前 `Biling` 仓库落地，而不是另起新项目
- 优先复用现有 Next.js 路由、鉴权、状态管理、接口封装模式
- 图片工作流作为独立能力域建设，不强行塞进当前文本 AI 面板交互

### 10.2 未决问题
以下问题需要在实施前进一步拍板，但不影响文档作为开发底稿使用：

1. 游客试用是否允许真实生成，还是只允许浏览示例
2. 免费额度最终采用“每日次数”还是“积分制”
3. 点数包用户是否享有无水印下载，还是仅 Pro 具备
4. 下载返回方式采用临时签名 URL 还是后端代理流
5. 图片水印采用生成后处理还是双资源存储
6. V1 是否接入真实支付，还是先只做订单与权益占位
7. 自定义模型供应商的配置入口采用环境变量、数据库配置，还是仅保留服务端代码适配

### 10.3 AI 开发代理执行提示
如果由 AI 编码代理直接实施，建议按以下顺序拆任务：

1. 补公共 DTO / 枚举 / 错误码
2. 建图片任务与资产 repository seam；有数据库软件时再落实体数据表
3. 完成业务服务的任务/资产/权益查询接口
4. 完成 AI 服务的任务提交与结果处理接口
5. 搭建前端工作台与资产中心页面
6. 实现局部重绘与扩图编辑页
7. 接入积分、会员、下载权限弹层
8. 补充联调与验收测试
